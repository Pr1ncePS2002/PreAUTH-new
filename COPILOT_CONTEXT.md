# Copilot Context — TPA Pre-Authorization Form Automation

This repo is a FastAPI backend + single-page frontend that:
1) accepts multiple patient documents (Aadhaar, policy card, estimates, notes, etc.),
2) runs OCR + key-value extraction using Gemini Vision,
3) auto-detects the most likely TPA form,
4) maps extracted keys into a **coordinate-based form schema** (JSON),
5) lets staff review/edit the mapped fields,
6) generates a filled PDF by drawing an overlay onto the blank TPA template.

If you’re a collaborator: tag this file and ask Copilot to “read this file + explain the flow / code.”

## Quick start (local)

1) Create a `.env` in repo root:

```env
GEMINI_API_KEY=...
GEMINI_MODEL=gemini-2.5-flash
# Optional (only needed if you use /auth/login for anything)
JWT_SECRET=dev-secret-change-in-production
```

2) Install deps:

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

3) Run the app:

```powershell
.\venv\Scripts\uvicorn.exe app:app --reload --port 8001
```

4) Open the UI:

- http://localhost:8001/ui

## Project layout (what to read)

- `app.py`
  - Main FastAPI app.
  - Implements workflow endpoints and session persistence.
  - Contains TPA template detection map (`TPA_TEMPLATE_MAP`) and some mapping helpers.

- `frontend/index.html`
  - Single-page provider portal UI.
  - Calls `/workflow/*` endpoints for the main user journey.

- `services/ocr_service.py`
  - OCR + structured extraction.
  - Default mode is Gemini Vision (`google-genai`).
  - Document AI mode is present but optional.

- `services/mapping_engine.py`
  - Maps OCR keys into schema field IDs using:
    - exact matches,
    - alias lookup (`config/field_mapping.json`),
    - fuzzy matching (RapidFuzz),
    - optional Gemini “suggest mapping” fallback.

- `services/form_engine.py`
  - Fills PDF templates using overlay rendering (ReportLab + PyPDF2).
  - Reads the analyzed schema JSON (coordinates by field).

- `services/his_service.py`
  - **Stub** HIS integration returning mock patients/documents/admission.
  - Backend endpoints exist for patient search/details.

- `templates/`
  - Blank TPA PDF templates. Many exist.

- `analyzed/`
  - Coordinate schemas (`*.json`) generated by Gemini/pdfplumber analysis.
  - Only some templates have schemas; UI only enables schemas that also have a matching template.
  - Files ending in `_gemini_raw.json` are treated as raw analysis artifacts and ignored by schema listing.

- `config/field_mapping.json`
  - Canonical field IDs + alias lists used by the mapping engine.

- Runtime folders (ignored by git; may contain PHI):
  - `uploads/` temporary uploaded docs
  - `output/` generated PDFs
  - `sessions/` persisted workflow sessions (raw OCR + mapped values)

## User flow (what the UI does)

The UI is a 3-step wizard in `frontend/index.html`:

### Step 1 — Upload + OCR
- User uploads multiple files per “document type” section.
- UI calls:
  - `POST /workflow/start` with:
    - `files`: all uploaded files
    - `document_types`: comma-separated list aligned with file order

Backend behavior (`app.py`):
- Saves uploads to `uploads/`.
- Runs OCR per file via `OCRService.extract()`.
- Merges extracted key-values into a single `raw_ocr_merged` dict (first-win merge: later docs do not overwrite earlier keys).
- Attempts TPA auto-detection from keys like `Insurance Company`, `TPA Name`, `insurance_company`.
- If a schema is available, performs a first mapping pass into `mapped_data`.
- Persists the session to `sessions/<session_id>.json` so it survives `uvicorn --reload`.

### Step 2 — Detect/Select TPA + Review/Edit
- If auto-detected, the UI selects that schema.
- Staff can manually choose another schema and click **Re-map**.
- UI calls:
  - `POST /workflow/{session_id}/remap` (form field `schema_name`)
  - `GET /workflow/{session_id}/schema-fields?schema_name=...` to get the schema field list

Backend remap behavior (`app.py`):
- Re-runs mapping against the selected schema:
  - MappingEngine pass (aliases + fuzzy)
  - A schema-label-based matching pass (fuzzy compare OCR keys to `field.label`)
  - Post-processing helpers:
    - `inject_hospital_data()` (hardcoded “Amrita Hospital” values for any hospital/provider fields present in schema)
    - `calculate_age_from_dob()` (if DOB exists, fills multiple age field variants across schemas)

Staff edits happen purely in the browser until “Generate PDF”.

### Step 3 — Generate + Preview + Download
- UI calls:
  - `PUT /workflow/{session_id}/data` with the final edited field dict
  - `POST /workflow/{session_id}/generate` with `template_name` + `schema_name`
  - Preview uses `GET /forms/preview/{form_id}` (embedded in an `<iframe>`)
  - Download uses `GET /forms/export/{form_id}`

## Schema format (coordinate JSON)

A schema file in `analyzed/*.json` typically looks like:

```json
{
  "form_title": "...",
  "total_pages": 6,
  "page_heights": {"1": 854.64, "2": 852.48},
  "page_widths": {"1": 612.0, "2": 612.0},
  "fields": [
    {
      "field_id": "patient_name",
      "label": "Name of the Patient",
      "type": "text_line",
      "page": 1,
      "coordinates": {"x": 250, "y": 359.3},
      "font_size": 9,
      "max_width": 345
    }
  ]
}
```

Coordinate system expectation:
- The analyzer captures `x,y` in a top-left-origin style (pdfplumber-like).
- The fill engines convert to PDF bottom-left coordinates using:
  - `y_pdf = page_height - y_schema`

## Mapping strategy (how OCR keys become field_ids)

There are two mapping layers working together:

1) Generic mapping via `services/mapping_engine.py`
- Input: OCR key-value dict.
- Output: `{field_id: value}`.
- Uses `config/field_mapping.json` aliases and RapidFuzz.
- Important behavior: alias index supports **multiple field_ids per alias**, then picks the first candidate that exists in the current schema’s `schema_fields`.

2) Schema-label-based fill-in (in `app.py` workflow endpoints)
- After the mapping engine runs, the workflow attempts to match any remaining OCR keys directly to the selected schema field `label` (fuzzy match).
- This helps catch schema-specific fields that aren’t represented in `field_mapping.json` yet.

## Adding a new TPA form (template + schema)

Minimum steps:

1) Add the blank PDF template to `templates/`.
   - Naming matters: if the template filename is `Some TPA.pdf`, the matching schema should be `analyzed/Some TPA.json`.

2) Generate the schema JSON into `analyzed/`.
   - Primary tool: `gemini_analyzer.py` (Gemini + pdfplumber hybrid analyzer).
   - There are also helper scripts in `scripts/` (e.g., batch coordinate fixes).

3) Update auto-detection keys (optional but recommended):
- Add synonyms to `TPA_TEMPLATE_MAP` in `app.py` so insurance company text can map to your template.

4) Improve mapping quality (recommended):
- Add/expand entries in `config/field_mapping.json`:
  - Add `aliases` for OCR labels you commonly see.
  - Add `gemini_description` if you plan to use Gemini fallback mapping.
